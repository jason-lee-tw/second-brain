## Key Concepts

- Ticket 1 goal: set up complete project foundation — Docker infrastructure, FastAPI skeleton, all SQLModel DB models, and Alembic migrations — so `docker compose up` starts all services, all 5 DB tables exist with correct schema, `GET /health` returns 200, and Alembic migrations run cleanly.
- Architecture: backend (FastAPI + SQLModel) and its Postgres share `app_network`; Phoenix and its own Postgres share an isolated `phoenix_network`. Backend reaches Phoenix via host port 6006 using `host.docker.internal`, a deliberate network-isolation security boundary — backend container never joins `phoenix_network` directly.
- DB schema is managed exclusively through Alembic migrations; SQLModel models are both ORM table definitions and the source-of-truth type definitions referenced by later tickets.
- Tech stack: Python 3.12, FastAPI, SQLModel, PostgreSQL 16 + pgvector, Alembic, pydantic-settings, Docker Compose, Arize Phoenix, pytest.
- Plan intended to be executed via superpowers:subagent-driven-development or superpowers:executing-plans, task-by-task with checkbox tracking.

### Task 4 — Database Models (TDD)
- Naming note (critical, referenced elsewhere in project CLAUDE.md): `DocumentChunk` Python attribute is `chunk_metadata`, mapped via `sa_column=Column("metadata", JSONB, ...)` to the SQL column `metadata` — done to avoid shadowing SQLModel/SQLAlchemy's class-level `metadata` attribute inherited from the declarative base. Rule for all future code: use `.chunk_metadata` in Python, `metadata` in raw SQL. This is documented as the *only* divergence from spec field names in the Self-Review section.
- `tests/unit/test_models.py` imports `ChatHistory, DocumentChunk, IngestedDocument, LearnedFact, ModelCorrection` from `second_brain.db.models` and instantiates each with representative field values (11 test cases total): `ChatHistory` (session_id str, thread_data dict defaulting to `{}`), `IngestedDocument` (uuid id, filename, content_hash, status, ingested_at, optional source_url defaulting to None), `DocumentChunk` (uuid id/doc_id, content str, embedding list[float] len 1024, chunk_index int, chunk_metadata dict defaulting to None), `LearnedFact` (fact str, embedding, source_session str FK, confidence float), `ModelCorrection` (original_answer, correction, root_cause strs, embedding, source_session FK) — includes an explicit test/doc-comment that embedding is computed from `correction` not `original_answer` (enforcement lives in the persistence node, not the model).
- Expected TDD red state: `ModuleNotFoundError: No module named 'second_brain.db.models'`.
- `db/models.py` — all 5 SQLModel `table=True` classes:
  - `ChatHistory` (`__tablename__ = "chat_history"`): LangGraph session state, keyed by UUID7 string that also serves as the LangGraph `thread_id`. Fields: `session_id: str` (PK), `thread_data: dict` (JSONB column, `default_factory=dict`, `nullable=False`), `created_at`/`updated_at: Optional[datetime]` (default_factory `datetime.now(UTC)`).
  - `IngestedDocument` (`__tablename__ = "ingested_documents"`): dedup record for ingested files/URLs. Fields: `id: uuid.UUID` (PK, default_factory uuid4), `filename: str`, `source_url: Optional[str]` (default None), `content_hash: str` (MD5 of raw file content, used to skip re-ingestion), `status: str` (`'processed' | 'failed'`), `ingested_at: Optional[datetime]`.
  - `DocumentChunk` (`__tablename__ = "document_chunks"`): a chunk from an ingested doc with embedding for RAG retrieval. Fields: `id: uuid.UUID` (PK), `doc_id: uuid.UUID` (FK → `ingested_documents.id`), `content: str` (chunk text with LLM-generated contextual header prepended), `embedding: list[float]` (`Vector(1024)` column, nullable), `chunk_index: int`, `chunk_metadata: Optional[dict]` (see naming note above), `created_at: Optional[datetime]`.
  - `LearnedFact` (`__tablename__ = "learned_facts"`): a user fact extracted from conversation for semantic retrieval. Fields: `id: uuid.UUID` (PK), `fact: str` (PII-scrubbed text), `embedding: list[float]` (`Vector(1024)`), `source_session: str` (FK → `chat_history.session_id`), `confidence: float`, `created_at`/`updated_at: Optional[datetime]`.
  - `ModelCorrection` (`__tablename__ = "model_corrections"`): a user correction to a model answer. Fields: `id: uuid.UUID` (PK), `original_answer: str`, `correction: str`, `root_cause: str`, `embedding: list[float]` (`Vector(1024)`, **encodes `correction` field, not `original_answer`**, per architecture decision — so cosine similarity retrieval surfaces the correct answer rather than the mistake), `source_session: str` (FK → `chat_history.session_id`), `created_at: Optional[datetime]`.
  - All embedding columns use `pgvector.sqlalchemy.Vector(1024)` — dimension 1024 is consistent across all 3 embedding-bearing tables (models, migration, and tests all agree).
- `db/session.py`: `engine = create_engine(settings.database_url, echo=False)`; `get_session()` is a FastAPI dependency generator that yields a `Session(engine)` and closes it on exit (context-managed).
- After implementation, expect 11/11 model tests pass, plus the 2 health tests = 13 total unit tests passing (`pytest tests/test_health.py tests/unit/ -v`).
- Commit message: `feat: add SQLModel table definitions and database session`.

### Task 5 — Alembic Setup and First Migration
- `alembic.ini`: `script_location = alembic`, `prepend_sys_path = .`, `version_path_separator = os`; `sqlalchemy.url` set to a localhost placeholder purely so the ini file parses — the real URL is overridden at runtime in `env.py` from `DATABASE_URL`. Standard logger/handler/formatter sections for root/sqlalchemy/alembic.
- Directory setup: `alembic/__init__.py`, `alembic/versions/__init__.py`.
- `alembic/env.py`: inserts `apps/backend/src` onto `sys.path` (computed as `Path(__file__).resolve().parent.parent / "src"`) so `second_brain` is importable from the alembic runner; imports `second_brain.config.settings` and imports `second_brain.db.models` purely for its import side-effect of registering all table metadata onto `SQLModel.metadata`; overrides `config.set_main_option("sqlalchemy.url", settings.database_url)`; `target_metadata = SQLModel.metadata`; defines both `run_migrations_offline()` (SQL generation without a live DB, using `literal_binds=True`) and `run_migrations_online()` (applies against a live connection via `engine_from_config` with `poolclass=pool.NullPool`); dispatches based on `context.is_offline_mode()`.
- `alembic/versions/001_initial_schema.py` — revision `"001"`, `down_revision = None` (first migration). `upgrade()`: runs `CREATE EXTENSION IF NOT EXISTS vector` first (must precede any VECTOR column creation), then creates all 5 tables in this order: `chat_history` (session_id PK, thread_data JSONB not-null with `server_default="{}"`, created_at/updated_at nullable timestamptz), `ingested_documents` (id UUID PK, filename, source_url nullable, content_hash, status, ingested_at nullable), `document_chunks` (id UUID PK, doc_id UUID with FK to `ingested_documents.id`, content Text, embedding Vector(1024) nullable, chunk_index Integer, `metadata` JSONB nullable — SQL column name, not `chunk_metadata`, created_at nullable), `learned_facts` (id UUID PK, fact Text, embedding Vector(1024) nullable, source_session with FK to `chat_history.session_id`, confidence Float, created_at/updated_at nullable), `model_corrections` (id UUID PK, original_answer/correction/root_cause Text, embedding Vector(1024) nullable, source_session FK to `chat_history.session_id`, created_at nullable). `downgrade()` drops tables in reverse dependency order (model_corrections, learned_facts, document_chunks, ingested_documents, chat_history) then `DROP EXTENSION IF EXISTS vector`.
- `tests/integration/test_migration.py`: module-scoped `db_engine` fixture connects to the real Postgres via `DATABASE_URL` env var and **skips** the whole module if the URL looks like a test placeholder (contains `test-api-key`, or lacks both `localhost` and `app_postgres`) — this guards against accidentally running integration tests with unit-test env vars. Test cases (9 total): all 5 tables exist (`test_all_five_tables_exist` via `inspector.get_table_names()`), pgvector extension enabled (`pg_extension` catalog query), `document_chunks` has expected column set including `metadata` (not `chunk_metadata` — this is the raw SQL/inspection name), `learned_facts` columns, `model_corrections` columns, `chat_history` columns, `document_chunks` FK references `ingested_documents`, `learned_facts` FK references `chat_history`, `model_corrections` FK references `chat_history`.
- Running the migration: `DATABASE_URL=postgresql+psycopg2://second_brain:secret@localhost:5432/second_brain alembic upgrade head` from `apps/backend/`, expecting log lines showing `Running upgrade  -> 001, Initial schema...`. Then run integration tests with the same DATABASE_URL, expecting 9 passed. Then `alembic check` expecting `No new upgrade operations detected.` (confirms models and migration are in sync, no drift).
- Commit message: `feat: add Alembic migration for all 5 database tables with pgvector`.
