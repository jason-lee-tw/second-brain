# OpenTelemetry + Arize Phoenix Tracing Implementation Plan

Source: docs/superpowers/plans/2026-06-16-ticket-2-otel-phoenix.md
Primary-Topic: otel-phoenix-tracing
Secondary-Topics: docker-compose-networking, fastapi-instrumentation

## Key Concepts

- Goal: instrument the FastAPI backend with OpenTelemetry and export traces to Arize Phoenix so a single `GET /health` request produces a visible end-to-end trace in the Phoenix UI at `http://localhost:6006`.
- Architecture: a `setup_tracing()` function initialises the global OTEL `TracerProvider` once at app startup, inside the FastAPI lifespan. `FastAPIInstrumentor.instrument_app(app)` wraps the app at module level so every HTTP request automatically gets a root span. A `trace_node(name)` decorator wraps async LangGraph node functions with child spans for future use.
- Tech stack additions: `opentelemetry-sdk>=1.29`, `opentelemetry-exporter-otlp-proto-http>=1.29`, `opentelemetry-instrumentation-fastapi>=0.50b0`, `arize-phoenix-otel>=0.8` (infers OTLP gRPC on port 4317), `pytest-asyncio`.
- File map of changes: `apps/backend/pyproject.toml` (add deps), `apps/backend/src/second_brain/config.py` (add `phoenix_collection_endpoint` field), `apps/backend/.env.template` (document `PHOENIX_COLLECTION_ENDPOINT`), new package `apps/backend/src/second_brain/observability/` with `__init__.py` and `tracing.py`, new test package `apps/backend/tests/unit/test_observability/`, `apps/backend/src/second_brain/main.py` (wire tracing + instrumentation), `docker-compose.yml` (add `extra_hosts` to backend service).
- `Settings` (in `config.py`) gains `phoenix_collection_endpoint: str = "http://host.docker.internal:4317"` — comment notes the backend reaches Phoenix via host port 6006/4317, and that on Linux Docker hosts `host.docker.internal` resolves only via `extra_hosts`.
- `.env.template` documents `PHOENIX_COLLECTION_ENDPOINT="http://host.docker.internal:4317"` with an override note for Linux hosts.
- `observability/__init__.py` re-exports `setup_tracing` and `trace_node` as the package's public API (`__all__`).
- `observability/tracing.py` defines:
  - `setup_tracing(phoenix_collection_endpoint: str, service_name: str = "second-brain") -> TracerProvider` — delegates to `phoenix.otel.register(project_name=service_name, endpoint=phoenix_collection_endpoint)`, returns the configured provider (also set as the global OTEL provider by `register`). Intended to be called once at startup inside the FastAPI lifespan.
  - `trace_node(name: str) -> Callable` — decorator factory for async functions. Wraps the function with `functools.wraps`, and inside the wrapper opens `tracer.start_as_current_span(name)` (tracer obtained via `trace.get_tracer(__name__)`) around the awaited call, so the span is a child of whatever span is active when the node runs — making it nest under the HTTP request span in Phoenix. Designed for wrapping LangGraph node functions, e.g. `@trace_node("orchestrator") async def orchestrator_node(state): ...`.
- Unit tests (`test_tracing.py`) use an `in_memory_tracer` fixture that swaps the global `TracerProvider` for one backed by `InMemorySpanExporter` + `SimpleSpanProcessor`, then restores the original provider after the test.
  - `TestSetupTracing`: asserts `setup_tracing()` calls the (mocked) `phoenix.otel.register` with `project_name` and `endpoint` matching the given args, defaulting `project_name` to `"second-brain"`, and returns whatever `register` returns.
  - `TestTraceNode`: asserts the decorator creates a span with the given name, preserves the wrapped function's return value, finishes the span before returning, and preserves `__name__` via `functools.wraps`.
  - `TestFastAPIInstrumentation`: `test_http_request_emits_span` builds a throwaway `FastAPI()` app, calls `FastAPIInstrumentor.instrument_app(test_app)`, hits it with `TestClient` (no lifespan trigger, so no real Phoenix connection needed), and asserts a span whose name contains `"GET /health"` is finished. Note: never call `FastAPIInstrumentor().uninstrument()` in teardown — it strips instrumentation globally from all apps including the real `main.app`, breaking other tests in the same session; instead let the local `test_app` be garbage-collected.
  - `test_main_app_health_emits_span` patches `second_brain.main.setup_tracing` (so no real Phoenix call happens), imports the real `second_brain.main.app`, uses `TestClient(app)` as a context manager (triggers the lifespan, thus the patched `setup_tracing`), hits `/health`, and asserts a `"GET /health"` span appears — validating the actual wiring in `main.py`, not just the pattern.
- `main.py` final shape: defines an `asynccontextmanager` `lifespan(app)` that calls `setup_tracing(phoenix_collection_endpoint=settings.phoenix_collection_endpoint)` then `yield`s; `app = FastAPI(lifespan=lifespan)`; immediately after app creation, at module level (not inside lifespan), `FastAPIInstrumentor.instrument_app(app)` is called so the OTEL middleware is present before any request arrives; a `GET /health` route returns `{"status": "ok"}`.
- TDD flow followed throughout: write/append failing tests first, run to confirm failure mode (e.g. `ModuleNotFoundError`, `AttributeError` for missing `setup_tracing` attribute on `main`), implement the minimal code, rerun to confirm all pass (final count: 8 tests passing across `TestSetupTracing`, `TestTraceNode`, `TestFastAPIInstrumentation`).
- `docker-compose.yml` change: add `extra_hosts: ["host.docker.internal:host-gateway"]` to the `backend` service. Rationale: Docker Desktop (Mac/Windows) auto-provides `host.docker.internal` resolution inside containers, but Linux Docker hosts do not — this mapping makes the hostname resolve to the host gateway IP on Linux without affecting Mac/Windows behavior. The `ollama-checker` service already has the same `extra_hosts` entry for the same reason.
- Full expected `docker-compose.yml` topology (as of this ticket): services `ollama-checker` (host network mode, curl-based healthcheck script, `extra_hosts` mapping), `app_postgres` (pgvector/pgvector:pg17, on `app_network`), `phoenix` (arizephoenix/phoenix:latest, port 6006 published, on `phoenix_network`, backed by `phoenix_postgres` via `PHOENIX_SQL_DATABASE_URL`), `phoenix_postgres` (postgres:17, on `phoenix_network`), `backend` (build context `./apps/backend`, `env_file` from `./apps/backend/.env`, on `app_network`, port 8000 published, depends on `app_postgres`, `extra_hosts` for host resolution). Networks: `app_network` and `phoenix_network` (both bridge). Named volumes: `app_postgres_data`, `phoenix_postgres_data`. Confirms the architectural rule that the backend never joins `phoenix_network` directly — it reaches Phoenix only via the host's published port.
- End-to-end verification steps (marked incomplete/unchecked in the plan — Step 2 of Task 5 was NOT checked off): `docker compose up -d`, `curl -s http://localhost:8000/health` should return `{"status": "ok"}`, then open Phoenix UI at `http://localhost:6006`, navigate to Projects → second-brain, and confirm a root span named `GET /health` appears with duration/status code in the waterfall. Troubleshooting notes: check `docker compose logs backend --tail=50` for OTEL export errors; common causes are `phoenix_collection_endpoint` overridden incorrectly in `.env`, the `phoenix` container not healthy (check `docker compose ps phoenix`), or on Linux `host.docker.internal` not resolving (verify via `docker inspect <backend-container>`).
