# Ticket 5 Memory System — Grilling Session Decisions

Source: docs/grilling-sessions/2026-06-26-ticket-5-grilling-decisions.md
Primary-Topic: memory-system
Secondary-Topics: database-connection-pooling, embedding-service

## Key Concepts

- Pre-implementation grilling session (2026-06-26) of the ticket-5 memory plan (`2026-06-16-ticket-5-memory.md`) and the second-brain design spec (`2026-06-16-second-brain-design.md`). PR #7 (`feat/005-memory`) was closed; PRs #8–11 introduced patterns conflicting with the original plan, prompting this re-grilling.
- Issues found in the original plan: `memory_retrieval.py` created its own `AsyncSessionLocal` (SQLAlchemy) — wrong driver for pgvector (critical, pgvector requires asyncpg); `asyncio.gather` used with a shared `AsyncSession` — not concurrency-safe (bug); function name mismatch `memory_retrieval_node` vs stub `retrieve_memory` (plan/code drift); `_last_human_content()` silently returned `""` on multi-modal input, inconsistent with the `get_str_content()` convention; `DATABASE_URL` env var read directly instead of via `settings.postgres_url` (inconsistency); `conflict_context: list[str]` too weak for the user to act on (API design gap); file map missing `db/pool.py`, `state.py` changes, `config.py` changes, `rag_retrieval.py` changes (incomplete).
- D1 — asyncpg Pool Architecture: move the asyncpg pool singleton into `second_brain/db/pool.py`, exposing `get_pgvector_pool()` and `shutdown_pgvector_pool()`. Both `memory_retrieval` and `rag_retrieval` query pgvector tables and should share this single pool rather than each owning an independent one — a node depending on another node (`rag_retrieval`) for its pool would be wrong coupling. Remove `_get_rag_pool()`/`shutdown_rag_pool()` from `rag_retrieval.py`; app lifespan imports shutdown from `db/pool.py`.
- D2 — MemoryPersistenceNode DB Writes: use the SQLModel sync `Session` (matching the `ingestion_agent` pattern) for writes; use the asyncpg pool for conflict-check reads. `LearnedFact` and `ModelCorrection` SQLModel models already exist in `db/models.py`; using them keeps ORM models and runtime code consistent, and `ingestion_agent` already establishes this pattern. Volume is only 1–3 inserts per turn, so the event-loop block from sync writes is negligible. Documented in `docs/codebase/003-database-management.md`.
- D3 — Conflict Threshold in Settings: add `memory_conflict_threshold: float = 0.85` to `Settings` in `config.py`, read from env var `MEMORY_CONFLICT_THRESHOLD`, so eval tooling can adjust the threshold at runtime without code deploys.
- D4 — Mutually Exclusive State Flags: `awaiting_correction` and `awaiting_conflict_clarification` are mutually exclusive; when conflict clarification is set, `awaiting_correction` is reset to `False`. Conflict affects data integrity while correction affects answer quality, so conflict takes priority.
- D5 — `conflict_context` Type Change: change `conflict_context` in `SecondBrainState` from `list[str]` to `list[ConflictContext]`, where `ConflictContext` is a `TypedDict` with fields `existing` (text of the existing fact), `existing_id` (UUID of the existing `learned_fact` row), and `new` (text of the proposed new fact). Rationale: the user must read the conflict and reply, so structured objects are actionable while bare strings are not. This is a breaking change — the `/query` response's `conflictContext` field changes from `list[str]` to `list[object]`.
- D6 — MemoryPersistenceNode Failure Strategy: per-fact retry (up to 3 attempts each), then raise, failing the entire node. Per-fact retry avoids duplicate writes on partial success; silent partial writes would create invisible memory gaps, which is worse than a surfaced error. The LangGraph checkpoint preserves `fact_updates` so a retry can re-attempt.
- D7 — Conflict Resolution Output: reuse `FactUpdate` with `conflicts_with: [existing_id]` rather than inventing a new TypedDict. `conflicts_with` was designed for exactly this; `MemoryPersistenceNode` already handles it by deleting IDs listed in `conflicts_with` and inserting the new fact.
- D8 — MemoryAgentOutput Schema: use a `MemoryCase` `StrEnum` (`FACT_EXTRACTION = "fact_extraction"`, `CORRECTION = "correction"`, `CONFLICT_RESOLUTION = "conflict_resolution"`) plus `with_structured_output` via LangChain-Anthropic. `MemoryAgentOutput` is a `BaseModel` with `case: MemoryCase`, `fact_updates: list[FactUpdate] = []`, and `correction_updates: list[CorrectionUpdate] = []`. The `case` field makes branching logic explicit and testable; `StrEnum` eliminates magic strings at comparison sites; `with_structured_output` avoids manual tool-call parsing. `MemoryCase` lives in `graphs/state.py`.
- D9 — Who Sets `awaiting_correction=True`: Synthesis sets both `is_uncertain=True` and `awaiting_correction=True` when `confidence < 0.7`, since Synthesis already holds `confidence` and setting both in one return dict is clean. Delegating this to `MemoryPersistenceNode` would give it a non-memory concern.
- D10 — Node Naming: use `memory_retrieval_node` (not `retrieve_memory`); graph wiring updated to `workflow.add_node("memory_retrieval_node", memory_retrieval_node)`.
- D11 — MemoryAgent Message Indexing: walk the `messages` list by type to find the last `HumanMessage` and the prior `AIMessage`, rather than using fixed negative indices — fixed indices break when `redact_outbound` appends the current AI answer before memory nodes run.
- D12 — Ollama Unavailability in `memory_retrieval_node`: fail hard (raise) rather than returning an empty `retrieved_memory`. Memory is load-bearing in this system, so silent degradation producing answers with no memory context would violate the system's purpose.
- D13 — Embedding utility: memory nodes must import `embed_text` from the existing `second_brain/services/embeddings.py` and must NOT create a new `utils/embedding.py`. `services/embeddings.py` already has a persistent `httpx.AsyncClient`, uses `settings.ollama_base_url` and `settings.embedding_model`, and has proper error handling — strictly better than the `utils/embedding.py` the original plan proposed (which would have added `get_embedding()` and `embedding_to_pg_literal()`, both unnecessary since asyncpg's registered vector codec accepts a Python list directly, no pg-literal string needed). The original plan was simply unaware `services/embeddings.py` existed.
- D14 — Embedding Unification Ticket (revised, deferred): a separate future ticket will extract `rag_retrieval._embed_query()` and unify it to use `services/embeddings.embed_text()`, to be planned together before implementation. Deferred under YAGNI — do not refactor `rag_retrieval.py` as a side effect of ticket 5, even though `_embed_query()` is now duplicated logic given `embed_text()` exists.
- D15 — LangChain-Anthropic scope limited to new memory nodes: the new `memory_agent.py` nodes created in this ticket use LangChain-Anthropic (`ChatAnthropic` + `.with_structured_output()`) to avoid manual tool-call parsing for `MemoryAgentOutput`. This does not change existing nodes — `ingestion_agent.py` continues to use the direct Anthropic SDK (`anthropic.AsyncAnthropic`) and is explicitly out of scope for this decision.
- D16 — Integration Test: use a real DB, following the same fixture pattern as `tests/integration/test_migration.py`, because cosine similarity, the pgvector `<=>` operator, embedding round-trip, and conflict-threshold behaviour are only meaningful when tested against a real database.
- New Ticket Required — Unify embedding utility: `services/embeddings.embed_text()` already exists as the canonical embedding helper, but `rag_retrieval.py` still has a duplicate inline `_embed_query()`. A new ticket should replace `_embed_query()` with `embed_text()` from `services/embeddings`; plan this ticket together before implementation (same as D14).
