# Second Brain Requirements

The approved Project Requirement Document (PRD) for the Second Brain — tech stack decisions, API contracts, agent roles, both LangGraph flows, database schema, and the 10 acceptance criteria that define "done".

## Key Concepts

- **Status and scope**: Approved, dated 2026-06-16. A personal "Second Brain" knowledge management system for mixed personal/work use — ingests local markdown files and web URLs, stores content for semantic retrieval, and maintains persistent memory of conversations/learned facts. Must demonstrate measurable improvement over a no-RAG baseline via rigorous evaluation. The full design spec lives at `docs/superpowers/specs/2026-06-16-second-brain-design.md`.
- **Tech stack decisions (with rationale)**: Python (required by assignment); FastAPI (lightweight, async-native, integrates with SQLModel); LangGraph for agent orchestration (required by assignment); PostgreSQL + pgvector (required by assignment); SQLModel + Alembic for ORM/migrations (SQLModel doubles as both DB models and FastAPI schemas); Arize Phoenix/OTEL for observability (required by assignment); embedding model `qwen3-embedding:0.6b` via Ollama (fully local, no API cost, dim=1024); `claude-haiku-4-5` for routing/web research/memory extraction; `claude-sonnet-4-6` for final answers and LLM-as-judge evals; Claude only as LLM provider (no Gemini); Tavily SDK for web search/crawl (required); Docker Compose for containerisation (required).
- **API endpoints and contracts**: `POST /query` (chat with the Second Brain) takes `{ "message": string, "sessionId": UUID7 | null }` and returns `{ "answer": string, "sessionId": UUID7, "confidence": float, "isUncertain": bool, "conflictDetected": bool, "conflictContext": [] }`. `sessionId=null` starts a new conversation; a returned UUID7 continues an existing one (it is the LangGraph `threadId` and chat-history key, stored in Postgres). `isUncertain=true` when confidence < 0.7, prompting the user to optionally correct the answer. `conflictDetected=true` when a newly extracted fact conflicts with an existing memory. `POST /ingest/file` processes `.md` files from `temp/pending-digest-docs/` and returns `{ "numberOfFilePassed": int, "failedFiles": [filenames] }`; each file is retried up to 3 times before moving to `temp/failed/`. `POST /ingest/url` crawls URL(s) via Tavily, saves the result as markdown, then ingests it via the same file-ingestion path.
- **Multi-agent pattern decision**: Single Supervisor Graph (LangGraph), with an LLM-powered orchestrator (not rule-based) reading the user query plus retrieved memory context to decide routing. RAG Retrieval and Web Research fan out in parallel via LangGraph's `Send` API when both are needed. Rejected alternatives: Hierarchical multi-graph (added indirection without benefit at this scope) and Agent-as-tool/ReAct (harder to enforce fan-out parallelism and a guaranteed synthesis step).
- **Two independent LangGraph graphs**: Query Graph (`SecondBrainState`, serves `POST /query`) and Ingestion Graph (`IngestionState`, serves `POST /ingest/file` and `POST /ingest/url`) share no runtime state — this keeps each graph's state schema clean.
- **Query Graph flow**: `messages[-1]` PII redaction (inbound) → memory retrieval (cosine similarity over `learned_facts` + `model_corrections`) → Orchestrator (`claude-haiku-4-5`, routes to rag/web/both/neither) → RAG Retrieval (pgvector top-k=5) and/or Web Research (`claude-haiku-4-5`, Tavily top-3), or pass-through → Synthesis (`claude-sonnet-4-6`, produces `final_answer` + `confidence` + `is_uncertain`; "neither" routing has a confidence floor of 0.5) → PII redaction (outbound, on `final_answer`) → Memory Agent (`claude-haiku-4-5`, fact extraction + correction detection, reading `messages[-2]` and `messages[-1]`) → conflict branch (sets `awaiting_conflict_clarification=True` and surfaces the conflict) or pass-through → Memory Persistence (writes `fact_updates` to `learned_facts`, `correction_updates` to `model_corrections`, generating embeddings via Ollama).
- **`SecondBrainState` fields**: `session_id`, `messages` (trimmed view; full history in checkpoint), `rag_results`, `web_results`, `retrieved_memory`, `routing_decision` (`Literal["rag","web","both","neither"]`), `final_answer`, `confidence`, `is_uncertain`, `awaiting_correction` (persisted across turns via LangGraph checkpointing), `awaiting_conflict_clarification`, `conflict_context`, `fact_updates`, `correction_updates`. Supporting TypedDicts: `RagResult` (content, score, chunk_index, metadata), `WebResult` (title, url, content), `MemoryItem` (id, fact, confidence, `type: Literal["learned_fact","model_correction"]`), `FactUpdate` (fact, confidence, `conflicts_with`), `CorrectionUpdate` (`original_answer` from `messages[-2]`, correction, root_cause).
- **Ingestion Graph flow**: files list → move file to `in_progress` + checkpoint state → Ingestion Agent (`claude-haiku-4-5`, chunk + embed + upsert to `document_chunks`) → on success move `in_progress` → `processed`; on failure check `retry_count < 3` — if yes increment and move to `retry_queue`, if no move to `failed` (terminal) → loop while `retry_queue` non-empty → return `numberOfFilePassed` + `failedFiles`. `IngestionState` TypedDicts: `FailedFile` (filename, error, retry_count) and `IngestionState` (files, in_progress, processed, retry_queue, failed). File folders: `temp/pending-digest-docs/`, `temp/processed/`, `temp/failed/`. Deduplication is by content hash (MD5) stored in `ingested_documents`; files with a matching hash are skipped on re-ingestion.
- **Document chunking strategy**: split on structural boundaries first (markdown headings H1/H2/H3 → blank lines → sentence boundaries), then apply a token cap. Markdown articles/notes target 512 tokens (max 1024, overlap 64); meeting transcriptions target 256 tokens (max 512, overlap 0); code fences are chunked atomically with no max/overlap. Each chunk gets a 50–100 token LLM-generated contextual-retrieval header prepended before embedding, based on Anthropic research showing 49–67% retrieval-failure-rate reduction. H1 > H2 > H3 heading hierarchy is stored as chunk metadata for filtered retrieval.
- **Memory system decisions**: Learned Facts are auto-extracted from every message when the user refers to themselves, embedded via Ollama, and conflict-checked via cosine similarity against existing facts before storing — on conflict, surfaced to the user pending clarification, then added/modified/removed. Model Corrections: Synthesis flags `is_uncertain=True` when confidence < 0.7; `awaiting_correction` persists across turns via LangGraph checkpointing; the Memory Agent detects a correction, extracts root cause, and stores it; if the user sends an unrelated query instead of correcting, `awaiting_correction` resets to `False`. Memory Retrieval: both `learned_facts` and `model_corrections` have `VECTOR(1024)` embedding columns, searched by cosine similarity at the start of every query; `model_corrections.embedding` deliberately encodes the `correction` field (not `original_answer`) so retrieval surfaces the correct answer, not the mistake.
- **PII guardrail decisions**: broad scope — names, emails, phones, physical addresses, national IDs, financial data, medical terms. Redacted with typed placeholders `[NAME]`, `[EMAIL]`, `[PHONE]`, `[ADDRESS]`, `[ID]`, `[CARD]`, `[MEDICAL]`. Applied at two points: inbound on `messages[-1].content` before any LLM node, and outbound on `final_answer` before it is appended to `messages`/persisted to `chat_history`.
- **Database schema (PostgreSQL + pgvector)**: `chat_history` (session_id UUID7 PK, thread_data JSONB, created_at, updated_at); `document_chunks` (id, doc_id FK → `ingested_documents.id`, content, embedding VECTOR(1024), chunk_index, metadata JSONB, created_at); `ingested_documents` (id, filename, source_url, content_hash TEXT MD5, status, ingested_at); `learned_facts` (id, fact TEXT PII-scrubbed, embedding VECTOR(1024), source_session UUID7 FK → `chat_history.session_id`, confidence, created_at, updated_at); `model_corrections` (id, original_answer, correction, root_cause, embedding VECTOR(1024) encoding `correction`, source_session, created_at). ORM is SQLModel + Alembic; pgvector integration via `pgvector-python`.
- **Observability decisions**: full distributed tracing at three levels per `/query` request — LLM call level (prompt/completion, token counts, latency), Agent/node level (which agents ran, order, duration, routing decision), and Request level (end-to-end HTTP-to-response trace). Phoenix stores trace data in `phoenix_postgres`, only reachable within `phoenix_network`. The backend exports via OTEL gRPC to Phoenix on host port 4317 and never joins `phoenix_network` — a deliberate security boundary. Docker network isolation: `app_network` = [backend, app_postgres]; `phoenix_network` = [phoenix, phoenix_postgres]. Linux Docker hosts need `extra_hosts: ["host.docker.internal:host-gateway"]` on the backend service.
- **Evaluation decisions**: dataset is hybrid — Claude generates ~100 Q&A pairs from ingested documents, curated down to ~30–50 (question + expected answer + expected source chunks). Retrieval-layer metrics: `context_precision`, `context_recall` via RAGAS. Answer-layer metrics: `faithfulness`, `answer_relevancy` via RAGAS with `claude-sonnet-4-6` as judge. Baseline comparison runs the same questions through (1) no-RAG (Claude only) and (2) the full RAG pipeline — RAGAS metrics must show measurable improvement of RAG over baseline. Run offline/on-demand, not part of CI. The `confidence < 0.7` uncertainty threshold is a starting value, calibrated during evaluation by measuring precision/recall of uncertainty flags against human-labelled ground truth.

## Acceptance Criteria

- AC-1: After a turn that extracts a user fact, `learned_facts` contains that fact with a valid embedding.
- AC-2: If a fact conflicts with existing memory, the API response includes a conflict notification and `awaiting_conflict_clarification=True`.
- AC-3: Given `awaiting_correction=True`, sending an unrelated query resets `awaiting_correction=False`.
- AC-4: Given `awaiting_correction=True` and a user correction, `model_corrections` contains root cause + correction with a valid embedding.
- AC-5: PII in user messages is redacted before reaching any LLM node.
- AC-6: PII in `final_answer` is redacted before being persisted to `chat_history`.
- AC-7: A file that fails ingestion is retried up to 3 times; on the 3rd failure it moves to `temp/failed/`.
- AC-8: A file matching an existing `content_hash` in `ingested_documents` is skipped on re-ingestion.
- AC-9: RAGAS `context_recall` and `answer_faithfulness` for the full pipeline are measurably higher than the no-RAG baseline.
- AC-10: `sessionId=null` creates a new LangGraph thread; subsequent requests with the returned UUID7 continue that thread.
