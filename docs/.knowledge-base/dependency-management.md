# Dependency Management

`uv` is the sole tool for adding, removing, and pinning dependencies across the workspace ‚Äî `uv.lock` is always regenerated by `uv`, never hand-edited.

## Key Concepts

- Adding a dependency: `uv add <package> --project apps/<service>`, run from the workspace root ‚Äî e.g. `uv add openinference-instrumentation-langchain --project apps/backend` when wiring in LangChain OTEL auto-instrumentation. Verify the addition via `grep "<package>" apps/<service>/pyproject.toml` rather than trusting the command silently.
- Removing a dependency: first confirm nothing still imports it (`grep -rn "<import-name>" apps/<service> --include="*.py"`), then `uv remove --directory apps/<service> <pkg1> <pkg2> ...`, then `uv sync --all-extras` plus the relevant test suite to confirm the workspace still resolves. Used to drop `langchain-anthropic` and `pandas` once the RAGAS collections migration made both fully unused.
- Dependency changes must be explicitly surfaced/flagged, never silently applied ‚Äî a project CLAUDE.md rule invoked directly when removing the now-unused `langchain-anthropic`/`pandas` packages.
- `uv.lock` and `.venv` are always regenerated, never hand-edited, after any version-pin or dependency-set change: `uv lock` regenerates the lockfile; `just clean-python && uv sync --all-extras` (or `uv sync --all-extras` alone after a removal) rebuilds the venv.
- Version-pin bumps (e.g. Python 3.12 ‚Üí 3.13) cascade through several declaration points across the workspace, all changed together in one task: `requires-python` in each workspace member's `pyproject.toml` (`apps/backend`, `apps/eval`), `ruff.toml`'s `target-version`, the Docker base image tag, and a new `.python-version` file ‚Äî the last of which is generated via `uv python pin 3.13`, never hand-written, so its content matches exactly what `uv` produces.
- Hard gate on lockfile resolution: if `uv lock` cannot resolve a compatible wheel (e.g. a cp313 wheel) for any existing dependency during a version bump, the upgrade is blocked outright ‚Äî no pinning an older sub-dependency, no `--no-binary` workaround, no silently skipping the package. The exact offending package and error must be reported and the work stops there.
- Dependencies flagged as most likely to hit compiled/platform-wheel resolution issues on a new Python version: `spacy`, `presidio-analyzer`, `presidio-anonymizer`, `psycopg2-binary`, `arize-phoenix-otel`, `langgraph-checkpoint-postgres`. All six resolved cleanly with cp313 wheels during the 3.12 ‚Üí 3.13 upgrade, so the hard gate did not trip.
- Version-range philosophy: prefer widening/using an existing pin over bumping it. RAGAS stayed pinned at `>=0.2.0,<0.5` during its collections-API migration because 0.4.3 (already inside the existing range) already shipped `ragas.metrics.collections` ‚Äî no version bump was needed to unlock the target API.
- Commits that change dependencies stage the affected `pyproject.toml`(s) and `uv.lock` together with the corresponding code/doc changes, using Conventional Commits (e.g. `chore(eval): remove langchain-anthropic and pandas, unused after RAGAS migration`, `config: upgrade Python 3.12 to 3.13`).
- Rollback for a version-pin bump is a pure code revert with no data/schema impact: `git checkout --` the version-declaration files, delete `.python-version`, then regenerate `uv.lock`/`.venv`/the Docker image against the prior version.
