Ticket 4 — Full /query Pipeline: PII, Routing, RAG, Web Research, Synthesis, Session Continuity

This diff wires up the complete POST /query flow for the Second Brain: an eight-node LangGraph that redacts PII on the way in and out, routes each query via an LLM orchestrator, fans out to RAG and/or web search, synthesizes an answer with a confidence score, and persists conversation state per-session in Postgres via LangGraph checkpointing. It was built as 12 TDD tasks (11 new modules plus the graph wiring and router registration), then hardened through two rounds of live-system verification and review that caught and fixed two release-blocking bugs and two important ones before merge.

Why

Ticket 4 is the backbone of the "Second Brain": every other capability (memory, corrections, ingestion feedback loops) depends on a working query pipeline that can hold a multi-turn conversation, decide where to look for an answer, and never leak PII either into the LLMs it calls or into what it stores. The spec called for this as a single LangGraph (SecondBrainState) kept deliberately separate from the ingestion graph, with session continuity implemented through LangGraph's own Postgres checkpointer rather than a bespoke chat_history table read/write path.

The task was decomposed into 12 small, mostly-parallel TDD units (state shape, PII service, PII nodes, memory stub, orchestrator, RAG node, web node, synthesis node, graph wiring, API schema/router, main.py registration, integration tests) so that independent nodes could be built and tested against a shared make_state() fixture without waiting on each other. The real engineering pressure showed up at integration time: two individually-correct-looking pieces (the checkpointer's connection pool defaults, and the outbound node's return shape) combined to break the system in ways that only appeared once you booted a real Postgres and ran an actual second turn — exactly the class of bug unit tests with mocks cannot catch, which is why a live-system verification pass was mandatory before calling this done.

State shape and message-history contract (Task 1)

SecondBrainState is the single TypedDict that every node reads and writes. Its most consequential design decision is that messages is Annotated[list[BaseMessage], add_messages] — LangGraph's built-in reducer that appends rather than overwrites. That one annotation is what makes "session continuity" possible at all: a node can return {"messages": [new_msg]} and LangGraph merges it into the existing checkpoint history instead of replacing the whole conversation.

The rest of the state (rag_results, web_results, retrieved_memory, routing_decision, confidence, is_uncertain, awaiting_correction, etc.) is plain overwrite-on-return, matching the per-turn scratch-pad role those fields play versus the cross-turn persistent role messages plays. A shared make_state() factory in tests/unit/conftest.py gave every subsequent node test a consistent, override-friendly state object to build on, which is what let 11 independent node/service modules be developed in parallel against the same contract.

Self-check

Why does messages use Annotated[list[BaseMessage], add_messages] instead of a plain list[BaseMessage]?
So LangGraph appends new messages to the checkpoint instead of overwriting the whole history on each node's return value.
Every node only returns the message(s) it produced (e.g. one HumanMessage or one AIMessage); the add_messages reducer is what merges that into the full, persisted conversation — required for AC-10 session continuity.
What problem does the shared make_state() conftest factory solve for this ticket's parallel-task structure?
It lets all 11+ node/service test files construct a valid SecondBrainState via keyword overrides instead of each hand-rolling the full 14-field dict, keeping tests decoupled from state-shape changes.
Tasks 2-12 ran largely in parallel; a shared factory with sane defaults meant a task only had to specify the fields it cared about, reducing merge friction across sibling worktrees.

PII guardrail — service and inbound/outbound nodes (Tasks 2-3)

services/pii.py wraps Presidio's AnalyzerEngine + AnonymizerEngine (spaCy en_core_web_lg under the hood) as a single redact_pii(text) -> str function, replacing detected entities with typed placeholders ([NAME], [EMAIL], [PHONE], [CARD], etc.). The analyzer/anonymizer are module-level singletons because spaCy model loading is expensive and should happen once per process, not per call.

nodes/pii_redaction.py applies this at the two boundaries the spec requires: redact_inbound scrubs messages[-1].content before any LLM node sees it (AC-5), and redact_outbound scrubs final_answer before it's persisted (AC-6). A subtlety in redact_inbound: it constructs a new HumanMessage with the same id as the original, so LangGraph's add_messages reducer replaces that message in place rather than appending a duplicate — the redacted version is what ends up in the checkpoint, the raw PII-bearing version never is.

The outbound node evolved during review (see F2 below): it now also builds an AIMessage from the same redacted string used for final_answer, computed exactly once, guaranteeing the checkpointed message history and the API response answer can never diverge — and that raw, unredacted PII is never even transiently present in messages.

Self-check

Why does redact_inbound preserve the original message's id when constructing the redacted replacement?
So LangGraph's add_messages reducer treats it as an update to the existing message rather than appending a new one, keeping the raw PII version out of the checkpoint.
add_messages matches on message id; reusing last_message.id causes an in-place replace instead of a duplicate append.
Why is redaction computed once inside redact_outbound and used for both final_answer and the new AIMessage, rather than redacting twice or redacting only one?
To guarantee the response returned to the API caller and the message persisted to the checkpoint are always identical and both fully redacted — no risk of one path leaking unredacted PII.
This was a review fix (F2): the original code never built an AIMessage at all, so computing the redacted string once and reusing it for both outputs closes both the missing-append bug and any redaction-divergence risk.
Why are AnalyzerEngine and AnonymizerEngine created as module-level singletons instead of per-call instances?
Because spaCy's en_core_web_lg model load is expensive; creating it once per process avoids paying that cost on every request.
services/pii.py comments explicitly flag this: "spaCy load is expensive, do it once."

Orchestrator routing and memory stub (Tasks 4-5)

nodes/memory_retrieval.py is intentionally a stub for this ticket — it always returns {"retrieved_memory": []}. This is a deliberate placeholder so the graph's shape (and everything downstream that reads retrieved_memory) is stable now and Ticket 5 can drop in the real pgvector similarity search against learned_facts/model_corrections without touching the graph wiring.

nodes/orchestrator.py uses claude-haiku-4-5 with with_structured_output to force a Literal["rag", "web", "both", "neither"] decision, given the current query plus whatever retrieved_memory context exists (currently always empty). This decision drives the fan-out in the graph: rag/web/both route to one or both retrieval branches in parallel via Send; neither skips retrieval entirely and goes straight to synthesis — important for latency on purely conversational turns.

Self-check

Why is retrieve_memory a stub that always returns an empty list in this ticket, rather than left out of the graph entirely?
So the node is wired into the graph now with the correct interface, letting Ticket 5 swap in the real cosine-similarity implementation with zero graph-structure changes.
The plan explicitly frames this as forward compatibility: "wires the node into the graph so Ticket 5 can drop in the real implementation with no graph changes."
What determines whether a query skips RAG/web retrieval entirely, and why does that matter for performance?
The orchestrator's routing_decision — when it's 'neither', the graph's conditional edge routes straight to synthesis, skipping both retrieval branches, avoiding unnecessary Ollama/Postgres/Tavily calls for purely conversational turns.
_route_retrieval in query_graph.py returns the string "synthesis" (not a Send) for the neither case, which LangGraph interprets as a direct edge rather than a fan-out.

RAG retrieval and web research nodes (Tasks 6-7)

nodes/rag_retrieval.py embeds the current query via the shared embed_text service (Ollama), then runs a cosine-similarity query (embedding <=> $1, i.e. pgvector's distance operator, converted to 1 - distance as score) against document_chunks, returning the top 5. It went through a real evolution during review: the plan's original version opened a brand-new asyncpg.connect() per request; the shipped version uses a lazily-created, module-level asyncpg.Pool (with register_vector wired through the pool's init callback so every physical connection gets vector type support), plus a shutdown() hook wired into main.py's lifespan.

nodes/web_research.py wraps the existing AsyncTavilyClient service, requesting top-3 results, with a fixed asyncio.sleep(1) before each call as a simple (not globally coordinated) rate limit — a deliberate design choice to keep the service a plain reusable Tavily wrapper while the node owns graph-specific throttling policy.

Both nodes independently need a bare postgresql:// DSN, since asyncpg/Tavily don't understand the SQLAlchemy +psycopg2 driver suffix present in settings.database_url — this suffix-stripping is currently duplicated in three places (flagged as deferred issue F5, not fixed in this ticket).

Self-check

What performance problem did the review flag in the original retrieve_from_rag, and how was it fixed?
It opened a brand-new asyncpg connection (full TCP + auth handshake) on every RAG-routed request; fixed by introducing a lazily-created, module-level asyncpg.Pool reused across requests, with shutdown() wired into the app lifespan.
This is finding F4 (important) in round-1 review — apps/backend/src/second_brain/nodes/rag_retrieval.py now uses _get_pool() / _pool instead of a per-call asyncpg.connect().
Is the web-research rate limiter (asyncio.sleep(1)) a true global rate limit across concurrent requests?
No — it's a per-call local sleep inside the node, not a shared/global limiter, so concurrent requests can still exceed 1 call/second in aggregate; this is a known, deferred minor issue (F6).
Each invocation of search_web sleeps 1 second before its own Tavily call, but nothing coordinates across simultaneous requests — it only throttles a single call's own pacing.
Why does register_vector get passed as the init callback to asyncpg.create_pool rather than called once after connecting?
Because register_vector needs to run against every physical connection the pool opens (not just the first), and asyncpg's pool init callback contract runs once per new connection.
Comment in rag_retrieval.py: "register_vector's signature is (conn, schema='public'), matching asyncpg's per-connection init callback contract — it runs once for every new physical connection the pool opens."

Synthesis node — confidence scoring and history trimming (Task 8)

nodes/synthesis.py calls claude-sonnet-4-6 with structured output (final_answer, confidence bounded 0.0-1.0, reasoning) over a prompt assembled from rag_results, web_results, retrieved_memory, and the last 10 messages (excluding the current query itself, via messages[-10:-1]) so the LLM has bounded context regardless of how long the conversation has grown.

The confidence handling has a deliberate special case: when routing_decision == "neither" (purely conversational turns with no retrieved context to ground the answer), the LLM's raw confidence is floored at 0.5 rather than left as-is — the spec's rationale is that a low-context conversational answer shouldn't automatically read as "the system doesn't know," but it also shouldn't claim false certainty, so is_uncertain (confidence < 0.7) can still end up true even after the floor.

Self-check

Why does synthesis trim conversation history to messages[-10:-1] rather than passing the full history or messages[-10:]?
To bound the prompt size regardless of conversation length, and to exclude the current query from the 'history' block since it's already included separately as the 'Current question'.
history_messages = state["messages"][-10:-1] deliberately drops the last element (the current query) to avoid duplicating it in both the history and query sections of the prompt.
If the LLM returns confidence=0.3 on a 'neither'-routed turn, what is the final confidence and is_uncertain value?
confidence becomes 0.5 (floored) and is_uncertain is True (0.5 < 0.7 threshold).
The neither-routing floor only raises confidence to a minimum of 0.5, it doesn't push it above the 0.7 uncertainty threshold, so low-confidence conversational answers still get flagged uncertain.
Does the confidence floor for 'neither' routing ever lower a high LLM-reported confidence?
No — it's a max(), so it only raises confidence when the LLM reported something below 0.5; a reported confidence of 0.99 stays 0.99.
confidence = max(confidence, _NEITHER_CONFIDENCE_FLOOR) — this is a floor, not a clamp or override.

Graph wiring, checkpointing, and the autocommit bug (Task 9 + F1)

graphs/query_graph.py assembles all seven nodes into a StateGraph(SecondBrainState): a linear prefix (redact_inbound → retrieve_memory → orchestrator), a conditional fan-out from the orchestrator using Send for parallel rag_retrieval/web_research branches (or a direct edge to synthesis for "neither"), both branches converging on synthesis, then redact_outbound → END. Session continuity is provided by AsyncPostgresSaver wrapping an AsyncConnectionPool, keyed by thread_id = session_id at invocation time.

The most consequential bug of the whole ticket (F1, blocking) lived in exactly one line here: AsyncConnectionPool(...) was created without kwargs={"autocommit": True}. Postgres refuses CREATE INDEX CONCURRENTLY — which AsyncPostgresSaver.setup() runs to create its own checkpoint tables — inside a transaction block. On any database that already had checkpoint tables (e.g. a dev machine reused across sessions), setup() is a no-op and the bug is invisible; on a genuinely fresh database it breaks graph construction outright. Task 12's integration tests actually hit this first and worked around it locally in the test fixture (pre-creating tables via an autocommit connection) while explicitly flagging the real fix for the review loop — which then made the one-line change (kwargs={"autocommit": True}) and deleted the workaround.

Self-check

What exactly caused build_query_graph() to fail on a fresh database, and why did it not show up in every test run?
AsyncConnectionPool defaulted to autocommit=False, and AsyncPostgresSaver.setup()'s CREATE INDEX CONCURRENTLY migration cannot run inside a transaction block — it only failed when the checkpoint tables didn't already exist, so a dev DB that already had them masked the bug.
This is finding F1 (blocking). Task 12 discovered it first via integration tests against a real Postgres and worked around it locally before it was root-caused and fixed for real in the review loop.
How does the graph decide whether to fan out to one, both, or neither of the retrieval branches?
Via _route_retrieval, the conditional edge function attached to the orchestrator node: it returns a list of Send(...) objects for 'rag'/'web'/'both', or the string "synthesis" for 'neither'.
Send enables LangGraph's parallel fan-out to independently invoke rag_retrieval and/or web_research with the current state; returning a plain string instead routes via a normal edge.
Why did Task 9 import Send from langgraph.types instead of langgraph.constants as the plan specified?
Because the installed langgraph==0.6.11 deprecates the langgraph.constants import path for Send.
Documented in decisions.md under task-9: a plan deviation driven by the actual installed library version, not a design choice.

API schema, router, and app registration (Tasks 10-11)

api/schemas.py gained QueryRequest (message, optional sessionId) and QueryResponse (answer, sessionId, confidence, isUncertain, conflictDetected, conflictContext) matching the spec's JSON contract exactly. api/routers/query.py implements POST /query: it generates a new UUID7 session id when none is supplied, lazily builds (and caches as a module-level singleton) the compiled graph on first request, invokes it with thread_id = session_id so LangGraph loads/continues the right checkpoint, and maps the result back into QueryResponse.

A second review fix (F3, important) hardened the error path: originally, any exception from graph.ainvoke() was serialized straight into the HTTP response body — a real risk of leaking the Postgres DSN or other credentials embedded in an exception's string representation to an API client. The fix logs the full exception server-side with exc_info=True and returns a generic, credential-free 500 message instead.

main.py was updated to register the new router and to call both query.shutdown() and rag_retrieval.shutdown() during the app lifespan's teardown, each wrapped in try/except-and-log so one failing shutdown doesn't block the other or crash the process. Two lazy singletons — the compiled graph in query.py and the connection pool in rag_retrieval.py — currently have no lock around their first-request initialization, a known race on concurrent cold starts (F7 and its sibling N1), deferred rather than fixed in this ticket.

Self-check

What information could leak to an API client before fix F3, and how was it fixed?
The raw exception string from graph.ainvoke() — potentially containing DSN/credential details — was returned directly in the HTTP response body; fixed by logging the full exception server-side (exc_info=True) and returning a generic 'Query graph error — see server logs' message instead.
apps/backend/src/second_brain/api/routers/query.py's except block now calls _logger.error(..., exc_info=True) then raises HTTPException(status_code=500, detail="Query graph error — see server logs") — no exception text reaches the client.
How does the router determine which LangGraph checkpoint/thread a given /query request continues?
It uses request.sessionId if provided (falling back to a freshly generated UUID7 otherwise) as session_id, then passes {"configurable": {"thread_id": session_id}} to graph.ainvoke — LangGraph loads that thread's existing checkpoint automatically.
session_id doubles as both the response's sessionId and LangGraph's thread_id, which is exactly what makes AC-10 (session continuity) work without a separate lookup table.
What known concurrency risk exists in _get_graph() and the RAG connection pool's _get_pool(), and why wasn't it fixed in this ticket?
Both are lazily-initialized module-level singletons with no lock, so two concurrent cold-start requests can each construct their own graph/pool; the loser's resources leak since only the winner's reference is retained. It was deferred as a minor issue (F7 / N1) rather than blocking merge.
Round-1 review flagged F7 for query.py's _get_graph(); round-2 review found the RAG-retrieval fix (F4) had introduced the identical race shape in a second singleton (N1) — both explicitly left for later rather than fixed now.
Why does main.py's lifespan wrap each shutdown call (query.shutdown(), rag_retrieval.shutdown()) in its own try/except?
So that a failure in one shutdown path doesn't prevent the other resource from being cleaned up or crash the app during teardown.
Each call is independently wrapped with except Exception: _logger.warning(..., exc_info=True), isolating failures per resource during shutdown.

Challenges & decisions

  • Task 12's integration tests discovered a real, previously-invisible production bug (F1): AsyncConnectionPool's default autocommit=False breaks AsyncPostgresSaver.setup()'s CREATE INDEX CONCURRENTLY migration on any genuinely fresh database — invisible on any dev DB that already had checkpoint tables. Worked around locally in the test fixture first, then root-caused and fixed for real (kwargs={"autocommit": True}) in the review loop, with the test workaround deleted and replaced by a genuinely isolated regression test.
  • Round-1 review found a second blocking bug (F2): synthesize_answer/redact_outbound never constructed an AIMessage, so assistant answers were silently never appended to the checkpointed message history — multi-turn context only ever contained Human turns, undermining conversation coherence and violating a later ticket's stated assumption that messages[-2] is the prior assistant response. Fixed by computing the redacted final_answer once in redact_outbound and returning both final_answer and a matching AIMessage, guaranteeing raw PII is never even transiently present in messages.
  • F3 (important): the /query error handler was forwarding raw exception text to API clients, a credential/DSN leak risk with no server-side logging for operators to debug from — fixed by logging server-side with exc_info=True and returning a generic client-facing message.
  • F4 (important): retrieve_from_rag opened a brand-new asyncpg connection per request instead of reusing a pool — fixed with a lazily-created shared asyncpg.Pool wired through register_vector's init callback, plus a shutdown() hook wired into main.py's lifespan (a follow-up pass caught that the hook was added but not initially wired in).
  • Task 3 deviated from the plan's literal no-PII test example ("What is the capital of France?") because Presidio's LOCATION recognizer genuinely flags country names as PII — an existing repo behavior surfaced by, but out of scope for, this task; the team swapped test text rather than modifying the shared PII service other in-flight parallel tasks depended on.
  • Task 5 deliberately hardcoded "claude-haiku-4-5" rather than adding a new Settings field, and Task 6 kept its DSN-stripping helper local to rag_retrieval.py rather than touching shared config.py — both explicit decisions to avoid merge conflicts with sibling tasks running in parallel worktrees.
  • Task 9 deviated from the plan's own smoke-test mock setup (which left AsyncConnectionPool.open() as a bare MagicMock instead of AsyncMock, causing a real TypeError against the actual async implementation) — corrected as a test-mock fix, not an implementation change.
  • Task 7 kept rate-limiting logic (asyncio.sleep(1)) in the graph node rather than the Tavily service, treating throttling as graph-specific policy while keeping the service a plain reusable wrapper — a deliberate layering choice, later flagged as an incomplete solution (F6) since it isn't a true global limiter.

Known risks & deferred items

  • N1 — rag_retrieval.py's lazily-created _pool singleton has no lock; concurrent cold-start requests can each create a pool, leaking the loser's connections (same race class as F7).
  • F5 — the +psycopg2 DSN-stripping logic is duplicated across rag_retrieval.py and api/routers/query.py instead of being centralized.
  • F6 — web-research rate limiting is a local per-call asyncio.sleep(1), not a true global/shared limiter, and adds fixed latency even without a burst of requests.
  • F7 — api/routers/query.py's _get_graph() lazy singleton has no lock, so concurrent cold-start requests can race into build_query_graph() more than once, leaking connection pools.
  • F8 — main.py imports the query router module under two separate names (query and query_router) purely to reach both .router and .shutdown().
  • MemoryRetrievalNode remains a stub returning an empty list for this ticket — real cosine-similarity retrieval against learned_facts/model_corrections is deferred to Ticket 5, so orchestrator routing and synthesis currently never see any long-term memory context.
  • Confidence threshold (0.7 uncertain) and the "neither" floor (0.5) are unvalidated starting points per the spec — calibration against human-labelled ground truth is explicitly deferred to the RAGAS evaluation phase (AC-9), not part of this ticket.
