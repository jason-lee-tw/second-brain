OTEL Tracing Coverage Fix: DB, Vector-DB, Model, and LangGraph Node Spans

Phoenix traces for the Second Brain backend only showed LLM calls made through LangChain — every LangGraph node boundary, every raw httpx/asyncpg/SQLAlchemy/psycopg call, and the checkpointer save/load were invisible black boxes. This change wires up four new OTEL driver instrumentors and finally activates an existing-but-unused trace_node() decorator across both LangGraph graphs, so a /query or /ingest/file request now shows a full nested waterfall: HTTP span → per-node spans → embedding/DB child spans.

Why

setup_tracing() (from an earlier change) passed auto_instrument=True to Phoenix's register(), which sounds like it covers everything but actually only auto-activates installed openinference-instrumentation-* packages — in this codebase, just the LangChain instrumentor. Raw driver calls (Ollama embeddings via httpx, pgvector reads via asyncpg, fact/correction writes via SQLAlchemy, LangGraph checkpoint save/load via psycopg) were never instrumented at all, and no generic OTEL instrumentor package for them was even installed.

Separately, trace_node() — a decorator built specifically to wrap a LangGraph node in a span — had zero call sites anywhere in src/. It existed, had tests, and did nothing, because its type guard (inspect.iscoroutinefunction(func)) rejects every node in this codebase: all nodes are singleton instances of BaseNode/BaseAgentNode subclasses with an async def __call__, not bare async functions, so the guard always saw False and raised.

The fix is two independent, additive changes with no business-logic impact: instrument the four raw drivers explicitly in setup_tracing(), and fix + wire up trace_node() at every add_node() call site in both graphs.

Letting trace_node wrap real nodes, not just bare functions

trace_node()'s guard used inspect.iscoroutinefunction(func), which is False for a class instance even when its __call__ is async def — and every production node (retrieve_from_rag, memory_retrieval_node, etc.) is exactly that kind of instance. The fix widens the guard to also check inspect.iscoroutinefunction(getattr(func, "__call__", None)), so an object with an async __call__ passes. The wrapper body itself (await func(*args, **kwargs)) needed no change — it already invokes __call__ correctly either way. Sync callables (plain def functions, or instances with a sync __call__) still correctly raise TypeError, because a built-in method-wrapper's __call__ also reports False from iscoroutinefunction.

A second, review-caught bug in this same function: functools.wraps(func) defaults to WRAPPER_UPDATES = ('__dict__',), which merges the wrapped callable's __dict__ onto the wrapper. For a stateful node instance (e.g. one holding self._agent), that silently leaked internal instance attributes onto the plain wrapper function returned by the decorator — a real (if currently harmless) break in encapsulation. Fixed by passing functools.wraps(func, updated=()) to skip that merge, with a regression test using a StatefulDummyNode to prove no leakage.

Self-check

Why did the original `trace_node` guard reject every real LangGraph node in this codebase, even though the nodes are all async?
Because the nodes are class instances with an async `__call__` method, not bare async functions, and `inspect.iscoroutinefunction(func)` returns False for an instance regardless of whether its `__call__` is async.
The fix adds `inspect.iscoroutinefunction(getattr(func, "__call__", None))` as an alternate check specifically to catch this case.
What did `functools.wraps(func)` leak onto the returned wrapper, and why didn't it show up for the original test helper (`DummyNode`)?
It merged the wrapped callable instance's `__dict__` (e.g. `self._agent`, `self.secret_state`) onto the wrapper function; the original `DummyNode` test helper had no `__init__` state, so there was nothing to leak.
`WRAPPER_UPDATES` defaults to `('__dict__',)` with no existence guard (unlike `__name__`/`__qualname__`, which skip missing attributes); the fix is `functools.wraps(func, updated=())`, verified with a new `StatefulDummyNode` regression test.

Instrumenting the raw drivers (httpx, asyncpg, SQLAlchemy, psycopg)

Four new dependencies (opentelemetry-instrumentation-httpx/-asyncpg/-sqlalchemy/-psycopg, all >=0.63b1) were added via uv add and wired into setup_tracing() right after register() returns the provider. Three of the four — httpx, asyncpg, psycopg — patch at the class level (the AsyncClient class, the asyncpg connection class, AsyncConnection.connect), so a bare .instrument() call works regardless of when clients/connections are actually constructed relative to setup_tracing().

SQLAlchemy is the exception, and this was the one thing the plan got wrong on first pass (caught by live verification, not by reading source first): a bare SQLAlchemyInstrumentor().instrument() only patches the create_engine() factory and Engine.connect() at the class level — it never retroactively attaches an EngineTracer to an engine object that already exists. Since db/session.py builds engine as a module-level singleton at import time (before setup_tracing() runs in the FastAPI lifespan), the bare call produced zero write spans in round-1 verification. The fix passes the existing engine explicitly: SQLAlchemyInstrumentor().instrument(engine=engine), importing engine from second_brain.db.session.

Self-check

Why does `HTTPXClientInstrumentor().instrument()` work with no arguments, while `SQLAlchemyInstrumentor().instrument()` needed `engine=engine` added?
httpx patches the `AsyncClient` class itself, so any client constructed later picks up instrumentation automatically; SQLAlchemy's bare call only patches the `create_engine()` factory and `Engine.connect()`, which never retroactively attaches a tracer to an engine object that was already constructed before `setup_tracing()` ran.
`db/session.py` builds `engine` as a module-level singleton at import time, before the FastAPI lifespan calls `setup_tracing()` — so the class-level patch alone can't reach it; only explicitly passing the existing instance (`instrument(engine=engine)`) attaches per-statement spans to it.
How was the SQLAlchemy `engine=` requirement discovered, and why does that matter for trusting the rest of this diff?
It was found via live verification against a running Phoenix instance in round 1 (write spans were missing), not by reading the instrumentor's source ahead of time — it's documented as a 'post-delivery correction' in both the spec and plan.
This is why the plan file itself carries blockquote corrections rather than being silently rewritten — the original plan text was wrong, and the fix was proven with real trace evidence, not just code inspection.

Wiring trace_node into the query and ingestion graphs

With trace_node() fixed, every async node is now wrapped at its add_node() call site: workflow.add_node("rag_retrieval", trace_node("rag_retrieval")(retrieve_from_rag)), and similarly for memory_retrieval_node, memory_agent, memory_persistence, orchestrator, web_research, synthesis in query_graph.py, and ingest in ingestion_graph.py. Wrapping happens at the graph-wiring site (not inside each node's own file) so the span name always matches the name it's registered under in the graph, and only two files need touching.

Two nodes are deliberately left unwrapped, and this was a real deviation from the plan's literal text, not a shortcut: pick_file_node (ingestion graph) and redact_inbound/redact_outbound (query graph, PII redaction) are all sync (def __call__, not async def) and do pure in-memory work with no I/O. trace_node only accepts async callables — wrapping any of them as the plan's original snippet literally instructed would raise TypeError at graph-build time and broke 4 pre-existing tests when tried. The implementer diagnosed this from pii_redaction.py's source before accepting the deviation, and the plan/spec were both updated with 'post-delivery correction' notes rather than silently rewritten.

Self-check

Why are `redact_inbound`, `redact_outbound`, and `pick_file_node` not wrapped in `trace_node`, unlike every other node in the two graphs?
They're synchronous (`def __call__`, not `async def`) and do pure in-memory work with no I/O (regex PII redaction / list slicing) — `trace_node` only accepts async callables, so wrapping them raises `TypeError` at graph-build time, and there's nothing worth tracing inside pure CPU work anyway.
This was caught as a genuine plan gap during Task 3, not implemented as originally instructed — the plan's literal snippet wrapped all 9 query-graph nodes and would have crashed `build_query_graph()` in production.
If you look at a Phoenix trace for a `/query` request after this change, would you expect to see spans named `redact_inbound` or `redact_outbound`?
No — their absence is expected behavior, not a bug to debug.
Both the plan and spec carry explicit 'post-delivery correction' notes warning verifiers not to treat their absence as a gap.
Why is each node wrapped at its `add_node()` call site in the graph files, rather than decorating each node at its own definition (e.g. inside `nodes/rag_retrieval.py`)?
So the span name can't drift out of sync with the name the node is registered under in the graph, and so the tracing concern stays in the two files that already own graph wiring instead of spreading across every node file.
The spec explicitly weighs this against making `BaseNode.__call__` concrete (would require touching ABCs plus ~9 node files) and against per-node-file decoration (risks span-name drift), and picks the `add_node()` site as cheapest and safest.

Test infrastructure: taming OTel's global-provider Once guard

OpenTelemetry's SDK protects the global TracerProvider with a Once guard: after the first set_tracer_provider() call in a process, every later call is silently ignored. Several new tests need to swap in an in-memory TracerProvider + InMemorySpanExporter per test to assert which spans got emitted — without resetting that guard between tests, the first test to swap providers permanently locks in its provider for the rest of the pytest session, breaking every other test that also needs to swap.

This was fixed with an autouse fixture (_reset_otel_tracer_provider) that saves and restores OTel's private _TRACER_PROVIDER and _TRACER_PROVIDER_SET_ONCE._done state around every test. It was initially added twice — once per test directory that needed it (test_graphs/conftest.py from Task 4, test_observability/conftest.py pre-existing) — and a review finding (I2) caught the duplication and hoisted it into the shared tests/unit/conftest.py, deleting both directory-local copies so it applies uniformly via pytest's fixture discovery.

Self-check

Why can't tests just call `trace.set_tracer_provider(my_test_provider)` and then restore the original provider in a `finally` block?
Because OTel's `Once` guard makes every `set_tracer_provider()` call after the first one in the process a silent no-op — so the 'restore' call in `finally` doesn't actually restore anything, and the test provider stays locked in for the rest of the session.
The fixture instead directly resets the private `_TRACER_PROVIDER` and `_TRACER_PROVIDER_SET_ONCE._done` attributes, which is the only way to actually unlock the guard between tests.
The fixture was originally added in two separate `conftest.py` files. What issue did that cause, and how was it resolved?
Near-duplicated fixture code across `test_graphs/conftest.py` and `test_observability/conftest.py`; a code review finding (I2) hoisted the single fixture into the shared `tests/unit/conftest.py` and deleted both directory-level copies.
Hoisting to the parent conftest means it applies to all subdirectories via pytest's fixture discovery instead of being maintained in two places that could drift apart.

Challenges & decisions

  • Deviated from the plan's literal Task 3 instructions by NOT wrapping `redact_inbound`/`redact_outbound` in `trace_node` — both are sync, no-I/O PII redaction; wrapping them as literally instructed broke 4 pre-existing tests and would crash `build_query_graph()` in production. Verified independently by reading `pii_redaction.py` before accepting the deviation.
  • Rewrote the plan's literal Task 3 test: the original drove the full graph via `graph.ainvoke()` against a mocked `AsyncPostgresSaver`, which crashed with cascading `TypeError`s from LangGraph's Pregel checkpoint loop (a generic Mock can't faithfully emulate `aget_tuple`/`get_next_version`/etc.). Replaced with direct `graph.nodes[name].ainvoke(state)` calls that isolate just the span-wrapping behavior being tested.
  • Root-caused a full-suite-only test failure (5 tests, invisible when running the new test file alone) to OTel's `Once` guard permanently locking in the first swapped `TracerProvider`; fixed with an autouse reset fixture, later deduplicated across conftest files (review finding I2).
  • Round-1 live verification found `SQLAlchemyInstrumentor().instrument()` (no args, as the plan specified) never emitted write spans — root-caused to the instrumentor needing an explicit `engine=` argument for engines already constructed before `setup_tracing()` runs. Fixed and reverified live in round 2.
  • Review finding I3 raised a plausible concern (psycopg checkpointer spans might have the same silent-no-op failure mode just found for SQLAlchemy) but live investigation confirmed psycopg's instrumentor patches `AsyncConnection.connect` at the class level (looked up fresh per call, unlike SQLAlchemy's one-shot factory wrap) — resolved as a false alarm with no code change, backed by a live Phoenix trace showing the expected spans.
  • Review finding R2-1 caught `functools.wraps(func)` leaking a wrapped callable instance's `__dict__` onto the trace wrapper (inert today, but a real encapsulation break) — fixed with `functools.wraps(func, updated=())` and a new stateful-instance regression test.
  • Two doc-correction findings (I4, then its follow-up R2-2) corrected stale claims in the spec and then the plan — the SQLAlchemy snippet and version number were wrong, and the `redact_inbound`/`redact_outbound` exclusion wasn't documented; both were fixed with additive 'post-delivery correction' blockquotes rather than silently rewriting the original text, to preserve the historical record of what the plan originally said versus what was actually shipped.

  • M3/M5: no structural test or guardrail prevents a future async node from being added to either graph without `trace_node` wrapping — relies on developer discipline.
  • M4: a vestigial try/finally `set_tracer_provider` restore remains in `test_ingestion_graph.py`, now fully redundant with the autouse conftest fixture.
  • M6/M12: span-capture test boilerplate (in-memory exporter setup) and dummy-node test helpers are duplicated across multiple test functions instead of hoisted into shared fixtures.
  • M7: a hand-rolled `base_state` dict in one test has drifted from the `make_state()` factory (missing `context_used`).
  • M8: the span-name string literal is retyped twice per `add_node()` call site (8 occurrences total) instead of derived once.
  • M9: the widened `is_async` guard in `trace_node` is not verified against a `functools.partial`-wrapped callable instance (no such node exists today, so untested).
  • M10: `trace_node`'s docstring/usage example still only shows the bare-function form, not the callable-instance form that all 8 production nodes actually use.
  • M11: no test covers the negative case of the widened contract — a sync callable instance (not just a sync bare function) should still raise `TypeError`.
  • M13: `_reset_otel_tracer_provider` is now autouse across all 226 unit tests (hoisted for DRY reasons) rather than scoped only to the ~3 modules that actually swap tracer providers.
